## 2b. ⚠️ `release.sh` is broken — and has been failing silently for two releases

**v0.3.8 is live**: https://github.com/sofirichof/studio-S/releases/tag/v0.3.8 — with
`Studio S.app.tar.gz`, its `.sig`, and `latest.json`. Verified end-to-end: fetching the app's
configured endpoint (`releases/latest/download/latest.json`) returns 0.3.8 and the tarball URL
answers HTTP 200. Sofia's installed app was left at **0.3.7 on purpose** so she can test the
in-app auto-update.

**The bug:** `src-tauri/tauri.conf.json` has `bundle.targets: ["app", "dmg"]`. The `dmg` target
runs Tauri's `bundle_dmg.sh`, which **uses AppleScript and dies headless**. Because `release.sh`
has `set -e`, the whole script aborts at that point — **before it ever writes the updater
artifacts or `latest.json`**. So `release.sh` produces nothing and exits non-zero.

**This is why releases 0.3.6 and 0.3.7 were never published** (latest GitHub release was v0.3.5
from Jul 9, while the local app was already 0.3.7). The script has been failing and nobody
noticed, because the failure looks like a DMG problem, not a release problem.

**Today's workaround** (manual, not committed into the script):
```bash
export TAURI_SIGNING_PRIVATE_KEY="$(cat src-tauri/.tauri-updater.key)"
export TAURI_SIGNING_PRIVATE_KEY_PASSWORD=""
npm run tauri build -- --bundles app        # skips the dmg target entirely
# then hand-assemble release-v0.3.8/ + latest.json, then: gh release create …
```

**The real fix — port the Gen Log solution.** Gen Log hit this exact bug and solved it: set
`tauri.conf` targets to `["app"]` only, and build the DMG in a standalone script using
**`hdiutil`** instead of AppleScript (staging dir + `Applications` symlink +
`hdiutil create -format UDZO`). See `~/Claude/Projects/Rotation 4 social - USB/gen-log/scripts/make-dmg.sh`
and memory `gen-log-app`. Then rewrite `release.sh` to build `app`, make the DMG via hdiutil,
and assemble `latest.json` — so one command works again.

**Consequence right now: there is NO 0.3.8 DMG.** Auto-update works (it only needs the tarball
+ sig + latest.json), but **a coworker doing a fresh install has nothing to download.** Fix
before any handoff to Lucas or the team.

when I'm clicking those options; the app does the translating for me."* **No new UI.**

**She already built one and didn't know it:** `dpTraits()` (`promptbuilder.html` ~L427) has ids
like `deakins` but only the *note* — "motivated naturalistic light, single soft key, deep
restrained shadows" — reaches the prompt. The DP's name never leaves the app. That's exactly the
pattern (and it's what the doctrine demands: no director/equipment names).

**She framed it as a pipeline, not content** — content rots in weeks; the pipeline is the asset:

1. **Intake** → claims land in `RESEARCH-LOG.md` first, never straight into the dictionary
2. **Verification** → source tiers (T1 API/vendor-docs/clicker → T2 dated primary → T3 rumour);

3. **Decay** → every entry carries a checked-date; monthly catalog diff catches deaths (the
   Sora lesson) — **automated, see below**
4. **Ship** → `dictionary.js` is the compiled *output* of 1–3

**Two specs are written and awaiting her review (she has NOT approved implementation):**

- `docs/superpowers/specs/2026-07-17-model-surface-two-axis.md` — split model from surface

  per-user AI Setup preference (only ~2 of the team use Higgsfield — do not hardcode it).

### 🔴 The video prompt — DO THIS FIRST. It is the actual complaint.

**This is the thing Sofia asked for in her second message of the day and it never got done.**
It was deferred four times into "out of scope — the bigger win" across two specs while @ tags,
the clicker, automation and a release all shipped around it. v0.3.8 went out with it still
broken. Do not let it slide again. If you only do one thing from this handoff, do this.

`compilePrompt()` in `src/promptbuilder.html` hardcodes the entire video prompt as:
```js
const moveTxt = { static:'locked-off, no camera move', push:'slow push-in', pan:'gentle pan',
                  tracking:'tracking move', handheld:'subtle handheld' }[s.move] || 'subtle motion';
video = 'Animate the still: ' + moveTxt + ', natural ambient motion, hold the framing.';
```

**Sofia's diagnosis — sharper than "it ignores the other fields":**

> *"video prompts aren't just the movement"*

She's right, and the problem is **architectural, not string formatting**. Reformatting the
template will not fix it. The builder is a **stills builder**: its six steps are Subject,
Environment, Look, Camera, Output, Review — every one of them describes **a frame**. The state
(`subject`, `environment`, `shot`, `angle`, `lens`, `move`, `depth`, `framing`, `dp`, `comp`)
contains exactly **one** time-based field: `move`. So the video prompt is only movement because
**movement is the only thing about time the app knows.** `subject` is a static noun phrase
("a woman pouring coffee"), not a beat — nothing captures *what happens* across 5–15 seconds.

The installed `higgsfield-seedance-prompt` skill is explicit that these are separate concerns:
an `ACTION` block ("events at the precision the shot needs — **camera motion and subject motion
stated separately**") and a `PERFORMANCE` block (acting beats: micro-pauses, eye-line,
restraint). The app has camera motion and nothing else.

**The fix is two parts — part 1 is not optional:**

1. **Add an action/beat input.** The builder must ask what *happens*. Minimum: one field on the
   Subject step ("What happens in the shot? e.g. *she pours, then looks up as the door opens*").
   Consider also a performance/acting note. Add to `builderKeys()` so it persists per shot.
   Without this, no template rewrite can produce a real video prompt — there's no data.
2. **Compose the video prompt from full state**, not one word. Per doctrine: subject action beat
   → camera move stated separately and *motivated* → ambient/environmental motion → light
   continuity → positive-form locks (same face, wardrobe, framing). Weave attached references
   (`weaveReferences` currently only touches `compiled.stills` — `compiled.video` is untouched,
   which is also why @ tags never reach the video prompt).

**Sequencing note — do NOT wait for the dictionary.** The dictionary spec lists this as
downstream, and that chain (worktree → model/surface → dictionary → video) is exactly how it got
deferred four times. It is independently valuable *now* and can be refactored onto the
dictionary later. The dictionary is blocked on a restructure that is itself blocked on a
worktree; this is not.

**Verify by running it**, not by reading the diff: two different shots must produce two
*meaningfully different* video prompts. Today every shot produces a byte-identical string apart
from one movement word — that's the regression test to write.
